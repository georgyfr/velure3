--------------------------------------------------------------------------------
  1. SYNTHESE GLOBALE
--------------------------------------------------------------------------------

Le document decrit un theme WooCommerce FSE (Full Site Editing) premium
cible les boutiques de complements alimentaires et nutraceutique.

Points forts du document :
  - Vision claire et differenciante sur le marche (IA nutritionnelle,
    configurateur de stack, transparence ingredient)
  - Architecture technique moderne (FSE, theme.json, React 18, CSS Layers)
  - Roadmap en 3 phases bien structuree (MVP, Premium, Enterprise AI)
  - Standards eleves : WCAG 2.2 AA, Core Web Vitals, GDPR, Schema.org

Points de vigilance :
  - Ambition tres elevee : certaines fonctionnalites (AR, NFT, Live Shopping)
    depassent le cadre d'un theme WordPress classique
  - Dependances a des services externes non specifies (API Claude, Redis,
    CDN, Mapbox, Rev.ai, etc.)
  - Budget estime 154 000 - 208 000 EUR sur 12 mois avec equipe senior
  - Incoherence partielle : le document cible PHP 8.3+ mais notre
    environnement dispose de PHP 8.4.21 (compatible, mais a noter)

--------------------------------------------------------------------------------
  2. FONCTIONNALITES CLASSIFIEES PAR PRIORITE
--------------------------------------------------------------------------------

Legende :
  [P1] = CRITIQUE — Indispensable au lancement (Phase 1 - MVP)
  [P2] = IMPORTANT — Differentiation commerciale (Phase 2 - Premium)
  [P3] = SOUHAITABLE — Avantage concurrentiel avance (Phase 3 - Enterprise)
  [P4] = OPTIONNEL — Nice-to-have, peut etre differe indefiniment

Faisabilite technique dans notre environnement :
  OK   = Faisable sans obstacle majeur
  PART = Partiellement faisable (limitations identifiees)
  DIFF = Difficile (necessite ressources/competences specifiques)
  N/A  = Non applicable dans notre contexte local

P1-10 | Performance Base

  Composants faisables :

    - Images WebP/AVIF avec fallback
    - Lazy loading natif (loading="lazy")
    - CSS minifie, JS defer/async

    - Fragment caching DIFFICILE sans Redis

  3. ANALYSE DE FAISABILITE TECHNIQUE PAR COMPOSANT
================================================================================

INFRASTRUCTURE :

  Redis / Object Cache Pro
    Statut : NON DISPONIBLE
    Detail : Pas d'acces root, Redis non installe.
    Impact : TTFB > 200ms, pas de fragment caching avance.
    Mitigation : Utiliser l'object cache WordPress natif (transient API),
               OPcache PHP (deja configure). En production, installer Redis.

  Nginx / LiteSpeed
    Statut : NON DISPONIBLE
    Detail : Environnement utilise Apache 2.4 sur port 8080.
    Impact : Pas de page caching Nginx/LiteSpeed.
    Mitigation : Utiliser .htaccess + mod_rewrite + mod_expires.
               En production, migrer vers Nginx si necessaire.

  CDN (Cloudflare R2 / BunnyCDN)
    Statut : NON APPLICABLE (environnement local)
    Impact : Assets servis localement.
    Mitigation : En production, configurer CDN pour assets statiques.

  Node.js / npm / Vite
    Statut : DISPONIBLE (user-space possible via nvm)
    Impact : Build React/TypeScript possible en user-space.
    Mitigation : Installer nvm + Node.js dans /home/z/.nvm

  API Claude (Anthropic)
    Statut : NON CONFIGURE
    Detail : Necessite cle API Anthropic.
    Impact : Toutes fonctionnalites IA (P2-02) bloquees sans cle.
    Mitigation : En Phase 1, utiliser systeme rule-based.
               En Phase 2, integrer API quand cle disponible.

  Mapbox
    Statut : NON CONFIGURE (payant)
    Mitigation : Utiliser Leaflet + OpenStreetMap (gratuit).

  WooCommerce Subscriptions
    Statut : NON INSTALLE (plugin payant ~279$/an)
    Impact : Fonctionnalites abonnement (P2-03) bloquees.
    Mitigation : Developper un systeme d'abonnement simplifie custom,
               ou installer le plugin quand licence disponible.

================================================================================
  5. RECOMMANDATIONS STRATEGIQUES
================================================================================

  1. DESACTIVER ELEMENTOR pour le developpement FSE
     NutriVitaX Pro est un Block Theme (FSE). Elementor utilise son
     propre systeme de templates qui entre en conflit avec les templates
     FSE. Garder Elementor pour d'autres projets, mais le desactiver
     quand NutriVitaX Pro est actif.

  2. COMMENCER PAR LE DESIGN SYSTEM (P1-02)
     C'est la fondation de tout le reste. Les tokens theme.json doivent
     etre definis en premier car tous les composants en dependent.

  3. SEPARER LE THEME DES PLUGINS
     Le theme ne doit pas inclure de logique metier complexe (IA,
     fidelite, abonnements). Celles-ci doivent etre des plugins
     compagnons ou des dependances a des plugins tiers. Le theme
     gere : templates, design, display, hooks WooCommerce.

  4. APPROCHE MVP REALISTE
     La Phase 1 (MVP) est realisable en environnement local dans un
     delai raisonnable. Les objectifs Core Web Vitals extremes
     (LCP < 1.2s) sont des cibles production, pas developpement.

  5. PLANNIFIER L'INSTALLATION NODE.JS
     Le build JavaScript (React blocks, Vite) necessite Node.js.
     Installer via nvm en user-space avant la Phase 2.
